Agent Context / Image / Profile — Unified Storage Matrix

Map Claude Code's on-disk context model onto the given Nexus two-layer design (/agents/{name}/ image + /proc/{pid}/ runtime), and track how sudocode & sudowork store the same things today (for dedup / migration). Draft for consensus — not yet merged into nexus-integration-architecture.

The frame (read first). Nexus's two namespaces are the given end-state, not an open question — same split an OS makes between an executable on disk and a running process (nexus-integration-architecture §2): image layer = /agents/{name}/ (config.toml · prompts/ · skills/ · memory/ · sessions/ — persistent, Metastore, the addressable identity) vs runtime layer = /proc/{pid}/ (status · agent→link · chat-with-me · workspace/ — ephemeral, stamped at start_session, reaped on exit; pid = AgentRegistry SSOT = the session id).

Primary key = agent-name (one name → many pids across worktrees). The workspace_hash you'll see under sessions/ is not a cwd-primary key — it is a per-repo sub-partition so one profile talking to many repos keeps its sessions sorted (§2.4). This is the distinction from CC/scode, whose primary key is the cwd.

This table's job: put CC's flat ~/.claude/projects/<cwd>/ model on each row, then read across — (1) does the Nexus end-state cover it? (2) how do sudocode & sudowork store it today, and what's dup to merge? CC is the reference we map from; Nexus is the target all three converge to.

Dependency direction (not a storage row — the rule that keeps the above honest). sudocode → nexus for VFS syscalls (KernelAbi sys_read/write/watch) and for services (A2A write; spawn-child via MAS::start_session). nexus → sudocode only through the SpawnTask<K> DI trait — sudocode-host registers a concrete adapter at boot; one vtable dispatch per session start, no compile-edge reversal. Self lifecycle state: sudocode reports up via a state_callback closure; ManagedAgentService (which owns AgentRegistry) writes the FSM transition — the kernel owns its task table, like a process not editing its own task_struct. Confirmed on sudocode origin/main v0.1.15: no AgentRegistry / ManagedAgentService type refs in sudocode; only KernelAbi + AgentDescriptor. CLI still runs StdFsBackend (seam dormant).

Open questions (for consensus)

Q1 workspace_hash semantics — confirm. agent-name is primary; workspace_hash is a per-repo session sub-partition under the profile (§2.4), not a cwd-primary key. Confirm this is the intended model (and that cross-agent search, Q3, is the only thing layered on top). (This replaces the earlier mistaken "remove workspace_hash" — the design is correct.)
Q2 Memory build-out. Which of the 5 missing memory features (extractMemories · session-memory · team-memory · auto-dream · daily-logs) does scode build, and in what order? And where does project-memory live when a workspace DT_LINKs several repos — at the primary mount root (≈ launch path)?
Q3 Cross-agent session search via DT_LINK /agents/{name}/sessions/ → shared /agents/sessions/ index? (depends on Q6)
Q4 Oversized tool-output offload (CC's tool-results/ + content-replacement) into scode — adopt? Path under /agents/{name}/sessions/<hash>/tool-results/?
Q5 Subagent parity. (a) Persist a per-subagent jsonl transcript (CC's agent-<id>.jsonl+.meta.json) vs today's md/json artifacts? (b) Keep subagent-fork at cache-prefix, or make it full-transcript like session --fork? (c) Move subagents to nexus-managed pids (sudocode → MAS::start_session)?
Q6 Workspace & DT_LINK. Confirm default-mount of the launch path as workspace[0] + multi-repo via workspace/{alias}→DT_LINK→mount_path. Cross-node is already free for federation-mounted targets (syscall re-entry inherits DT_FILE try_remote_fetch). Decision: constrain link targets to federation-mounted paths (ship now, zero change), or invest in owner-discovery for arbitrary remote targets (separate design)?
Q7 Standalone resume pointer. Adopt CC's bridge-pointer.json (mtime-heartbeat + cross-worktree-freshest) for the offline scode case? (AgentRegistry is SSOT when nexus is present; standalone has no stored pointer today.)
Q8 Migration sequencing. Order of: (a) wire scode's dormant KernelFsBackend seam (flat .scode/sessions/ → layered /agents/{name}/); (b) collapse sudowork messages → a view over the engine JSONL keyed by acpSessionId; (c) build the missing memory features. Which first?
