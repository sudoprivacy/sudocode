Image handling should be NON-user-facing

Design draft for review · 2026-06-29 · Revised 2026-06-30 (rev2 — 16 ShareOne comments folded in) · Owner: sudowork-win-pc-0 · Tracking row: sudo-code-roadmap §9 "Image too large / wrong-model should be NON-user-facing"

The user pastes an image (or an agent screenshots a page, or the conversation embeds a generated chart) in any sudowork-driven flow. The image always ends up handled — either as a real vision input or as a transparently-spliced text description. The user never sees:
  • "图片太大了，请压缩后再发送" — size errors
  • "当前模型不支持图片分析" — wrong-model tips
  • A stuck processing spinner
  • A "[Image #N was too large to send]" placeholder
Any of those today is a leak from infrastructure constraint up to user task interrupted. This design closes them.

Real-world symptoms in scope

  1. User attaches a large photo in a chat with Claude/Gemini-family models → today: 进二 hit single_request_too_large; sudowork dropped chat history silently. Already fixed end-to-end via PRs PR-A/B/C/D/E (image-attach preflight + breaker + log line). Design covers the residual cap-derivation question.
  2. Agent uses ai-dev-browser to view a page containing large images/animations (e.g. a ShareOne-generated page). The agent's screenshot or DOM-export is what reaches the LLM; if the screenshot is too large, the user sees "图片太大了，请压缩后再发送" on a follow-up action they didn't trigger — the most user-confusing variant. Comment #4 (作者 e6fc604a) flagged this directly.
  3. User picks a non-vision model and the conversation already has an image attached → today: sudowork emits "当前模型不支持图片分析" and aborts the turn. The user has to remember to swap models OR delete the image.

Architecture decisions

Decision 1 — Cap SSOT: ACP capability negotiation (per-backend) comments #1, #2, #5, #13

Each ACP backend advertises its preflight capability via the ACP _meta field on the standard initialize response. Sudowork queries; never hardcodes.

Method placement (comment #2 verification): the ACP spec from agentclientprotocol.com (the Agent Client Protocol — maintained as an open community spec, lead maintainer Sergey Ignatov; not a Google protocol — common misnomer) defines:

  • The lifecycle method initialize — its response already carries agentCapabilities.promptCapabilities.image: boolean. That's binary (yes/no), too coarse for byte/dim/target.
  • The _meta field — formally specified in RFD: Meta Field Propagation Conventions as the canonical extension point on all protocol types, with { [key: string]: unknown }. Reserved root-keys are traceparent, tracestate, baggage; vendor namespaces (like ours) live under custom keys.

So our extension goes on initialize response, under _meta.imageCapability:

initialize response (ACP standard; _meta extension):
  {
    "protocolVersion": 1,
    "agentCapabilities": {
      "promptCapabilities": { "image": true, ... }, // existing boolean
      ...
    },
    "_meta": {
      "imageCapability": {                    // sudowork-defined extension
        "maxBytes": 5242880,                  // 5 MB — backend's hard ceiling
        "maxDimension": 8000,                 // 8000 px
        "downsampleTargetBytes": 524288,      // 512 KB — backend's recommended target
        "autoHandlesOversized": true,         // backend will downsample on receive
        "autoHandlesWrongModel": true         // backend will VLM-route for non-vision models
      }
    }
  }
  • sudocode: implements + advertises the full imageCapability.
  • Claude Code / Codex / Qwen: do not advertise _meta.imageCapability (standard ACP only).
  • Comment #1 — graceful degradation (HARD RULE): absence of _meta.imageCapability means "backend doesn't speak this extension"; sudowork falls back to its current defaults silently, no throw, no warning, no user-visible artefact. The ACP session must connect successfully even if the backend predates this spec.

Source of cap data (comment #5): sudocode's imageCapability::capability() reads from sudorouter's model-capability table, NOT a hardcoded constant. The sudorouter side already authoritatively knows each model's image_max_bytes / image_max_dimension / vision_supported. Sudocode's job is to relay the active-model cap, not invent caps. Implementation note: if sudorouter's table is missing a model, sudocode emits a conservative default + logs (so sudorouter AI can add the entry). Coordination ask: post a request to sudorouter AI to expose the per-model image-capability fields in the existing model-capability response (already a sudorouter ↔ sudocode contract).

Comment #13 — is _meta.imageCapability schema worth debating: no. _meta is the spec-blessed extension point, the field names are descriptive, breaking changes are easy (just bump a sub-version inside _meta). Dropped as an open question.

Why this satisfies SSOT: each backend is its own authority on its caps + behaviour. Sudowork is a faithful client. No cross-repo hardcoded constants that drift.

Decision 2 — VLM-route logic lives in BOTH sudocode and sudowork, but only ONE fires per turn comment #7

  • Sudocode (Rust, in runtime crate): new module vlm_describe.rs. When push_images decides not to ship the image as ContentBlock::Image (because: a) decoded image fails preflight at q25, or b) active model isn't vision-capable per sudorouter cap), it calls VLM endpoint → substitutes ContentBlock::Text { "[Image #N: <description>]" }. Sudocode owns its own behaviour; advertises autoHandlesOversized=true, autoHandlesWrongModel=true.
  • Sudowork (TS, sudowork-side fallback): when active ACP backend advertises autoHandlesWrongModel=false (or omits the capability entirely), sudowork keeps the existing callChatCompletionsWithImage flow but promotes it to transparent fallback — fires automatically when chat-attach image hits a non-vision chat model, splices text into the user message, dispatches normally.

DRY + SSOT guarantee (comment #7): the _meta.autoHandles* flags are how sudowork knows whether to fire its fallback. If the backend advertises true, sudowork hands off and does nothing. If false or absent, sudowork fires its own fallback. Exactly one VLM call per turn. The single source of "who handles it" is the _meta.imageCapability response on session init — no parallel decision tree.

Decision 3 — Caps are model-driven, not heuristic comment #12

The current LOW_IMAGE_TARGET_PATTERN = /gemini|claude/i → 128 KB in sudowork was an empirical override from earlier commits (1e52d407, 938cbdd8). Comment #12 is right: 128 KB is artificially conservative — Gemini is SOTA on vision and Claude is also strong; both can accept much larger images.

Replace heuristic caps with model-driven caps:

  • Default cap = the active model's documented limit, as advertised by sudorouter's model-capability table and surfaced through sudocode's _meta.imageCapability.
  • Sudowork's tryReadAsImage queries the ACP session's advertised downsampleTargetBytes / maxDimension first; falls back to current defaults only when the backend omits the _meta (per Decision 1).
  • If a user is explicitly cost-sensitive, expose a single user-side opt-in (e.g. settings.image.economyMode: true) that caps to the lower of (model limit, 128 KB). Default OFF.
  • The hard fallback IMAGE_MAX_BASE64_SIZE = 5 MB stays — it's a sanity limit, not a heuristic.

Concretely the change in src/common/imageUtils.ts:

  • LOW_IMAGE_TARGET_PATTERN + the 128 KB override → removed
  • IMAGE_TARGET_RAW_SIZE default 512 KB → kept as "no-capability fallback default" only
  • tryReadAsImage(filePath, ...) picks maxBytes = sessionCap?.downsampleTargetBytes ?? IMAGE_TARGET_RAW_SIZE

Decision 4 — Keep dimension cap (1280 px) sudowork-side

Sudowork already constrains dim to 1280×1280 (IMAGE_MAX_WIDTH/HEIGHT). Smaller than sudocode's 8000. After unification: sudocode advertises 8000 in capability; sudowork applies min(client default, advertised value). Pre-existing client-side latency optimisation kept.

Decision 5 — ai-dev-browser screenshots are part of the same SSOT comment #3, #4

Comment #3 asks the right question: ai-dev-browser is sudoprivacy-owned, sudowork uses it as a tool, and it produces images that flow through the same LLM-input pipeline. Today its screenshot resolution/JPEG-quality presets are local-to-itself — disconnected from the active model's cap.

Decision: extend the SSOT to ai-dev-browser. The ACP _meta.imageCapability values flow into sudowork's tool-invocation layer; when sudowork calls ai-dev-browser's screenshot tool, it passes the active session's downsampleTargetBytes / maxDimension as an arg — ai-dev-browser obeys.

Coordination ask (per memory cross_ai_routing): post to ai-dev-browser AI requesting:

  1. A typed image_cap field on the screenshot tool's input schema: { maxBytes?: number, maxDimension?: number }.
  2. Implementation: when image_cap is provided, the screenshot pipeline downscales/recompresses to fit; absence means existing behaviour.

This directly resolves comment #4 (the ShareOne-page screenshot scenario): when the agent screenshots a ShareOne page, the screenshot is already capped to the active model's accept-size, so the "图片太大了" prompt never surfaces.

VLM model selection (sudocode-side + sudowork-side) comment #8

Mirrors the existing /image analyze path:

  • Credentials: readSudorouterCredentials() (src/process/bridge/imageGenerationBridge.ts:91) — already reads from sudocode.json → auth_modes.proxy.sudorouter, falling back to legacy sudoclaw.json.
  • Model selection logic — TWO sub-cases:
    • Oversized-but-vision-capable (sudocode side): the active model CAN take images, just not THIS one's size. Use the active model as the VLM (mirrors resolveChatModel()). One round-trip extra. Output is a text description spliced into the same turn.
    • Wrong-model (sudowork-side OR sudocode-side): the active model can't take images at all. Use a designated vision-analysis model — parallel to resolveImageConfig() (the create-image path's model). Add a new sudoclaw.json slot: imageAnalysis: { model: "gemini-2.5-flash", baseUrl, apiKey }. Default: gemini-2.5-flash (cheap + fast + strong vision; the same default the /image analyze path effectively uses today via primary chat model when that's Gemini).
  • Prompt template: "Describe this image concisely (1-3 sentences). Focus on: text content visible in the image, key UI elements, and anything the user is likely to be asking about." Mirrors the existing analyze prompt shape.

Latency + token cost (the real numbers) comments #9, #10, #11

Latency (comment #9 — original framing was wrong)

The doc previously claimed "~0.5–2s extra latency". That's only true in a hypothetical where the LLM could have taken the image natively but we routed through VLM anyway. In the actual fallback path the LLM CANNOT take the image (either oversized at q25, or model isn't vision-capable) — without the fallback, the baseline is an error message + user re-uploads or swaps models, which is dramatically slower wall-clock than +1 VLM round-trip. So the correct framing is:

  • Fallback path (this design): 1 VLM round-trip (~0.5–2s) + 1 LLM round-trip (same as before). Wall-clock change vs status-quo "show error message": strictly faster for the user (no manual recovery).
  • NOT proposed: routing through VLM when the LLM could take the image natively. That would add latency for no benefit.

Token cost (comment #10)

Empirically, VLM input tokens are billed similarly to LLM image-input tokens at the same provider tier (both account for image pixels via the same vision-encoder; e.g. on the Gemini Flash family the per-image input tokenisation is ~258 tokens per 384×384 tile regardless of which API entry-point). The VLM's text output (~50–150 tokens) becomes part of the LLM's text input. Net comparison:

  • Native vision path (when available): LLM(image_tokens + rest)
  • Fallback path (this design): VLM(image_tokens) + VLM-text-out (~100 tokens) → LLM(100 text_tokens + rest)

Roughly: fallback ≈ native + 1 small VLM text generation. Cost-wise it's a small surcharge per oversized/wrong-model image. Cost is NOT the reason to avoid the fallback; UX is the reason to embrace it.

VLM "quality loss" reframed (comment #11)

Comment #11 is right: VLM is the expert on reading images, not a degraded substitute. The actual loss is in the natural-language transduction — turning rich visual context into 1–3 sentences of text loses spatial layout, exact colour values, fine OCR positioning. So:

  • For "what does this image generally show / read me the text" tasks: no quality loss (VLM ≥ generic LLM on raw image comprehension).
  • For "click pixel (412, 198) in this image" tasks: meaningful loss (text can't preserve pixel coordinates).
  • The fallback is only ever invoked when the LLM cannot take the image — so the comparison is against "error message + manual recovery", not against perfect native vision.

Wrong-model handling — concrete wording comment #14

"Wrong-model" = the user's active chat model is text-only (e.g. gpt-4o-text, claude-3-haiku-text) but the conversation includes an image attachment. Today sudowork's AcpAgent.ts:986 emits "当前模型 X 不支持图片分析" and aborts the turn (scode backend only). After this design:
