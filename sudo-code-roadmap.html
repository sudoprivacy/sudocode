What sudo code is

sudocode (binary: scode) is a Rust-native ACP engine for coding agents — the hacker-facing CLI half of the sudo* family.

sudoworksudocode
AudienceNon-technical end usersDevelopers, hackers, machines
SurfaceGUI / ElectronCLI / headless / ACP
DefaultsSafe, hand-held, friendly copyComposable, terse, full power
Relationshipsudowork uses sudocode as one of its execution engines

The two are tuned for different audiences. The same capability can land safely-defaulted in sudowork and exposed as a full knob in sudocode.

North star

  • Rust-native — single binary, deterministic shutdown, lean footprint.
  • Model-agnostic — Anthropic, OpenAI, xAI, Gemini, OAuth subscriptions, arbitrary proxy backends.
  • Headless-first — ACP over stdio and WebSocket; embeddable as "agent as a service".
  • Safe by design — explicit permission modes plus a Linux user-namespace sandbox.

Goal 1 · Lock the baseline — e2e ≥ 90%

scode CLI e2e coverage reaches and stays at ≥ 90% of the scode-native testable feature surface, green on every main commit of sudocode CI.

Test infrastructure — two layers, one of them counts

LayerPurposeCounts toward 90%?Source
pty-expect PTY harness Human-fidelity. Drives scode through a real PTY (Unix /dev/pty, Windows ConPTY) and asserts against either the raw stream or a VT100-rendered view. Catches what a real user sees: REPL UX, signals, tab completion, ANSI redraw, cursor placement. Yes — this is the coverage measure. sudoprivacy/pty-expect
Mock parity harness Fast, in-process. Scripts a mock Anthropic backend and asserts on the byte stream scode produces. Catches API integration / tool dispatch / streaming regressions without any TTY. No — it is a CI gate, not an e2e measure. docs/mock-parity-harness.md
Coverage rule: a feature is Covered only when pty-expect exercises it end-to-end on main CI. The mock parity harness still runs on every PR as a required gate and catches API / dispatch / streaming regressions early — but it tests the lower half of the stack ("does scode dispatch and stream correctly?") and not the upper half a user actually sees ("does the REPL render it correctly, does Ctrl+C interrupt cleanly, does the prompt redraw without flicker?"). Those upper-half failures are what the 90% goal is about, so only the PTY layer earns coverage credit.

This makes the 90% target a strictly higher bar than the previous "at least one harness exercises it" rule. It mirrors the same anthropomorphic-fidelity argument that drives ai-dev-browser's real-Chromium + real-mouse stance for sudowork: the only way to catch the failure modes a human will hit is to exercise the same surface the human exercises.

TestEnv dual-mode: tests/common/mod.rs provides TestEnv, the single entry point for all PTY tests. It supports mock (MockAnthropicService, CI-safe) and live (SCODE_TEST_BACKEND=live, real API). All PTY tests MUST go through TestEnv.

PTY test principles

  1. Live quality first. Every assertion must be meaningful against a real API. If an assertion only verifies the mock's canned response (e.g. expect("roundtrip complete")), it's testing the mock, not scode — delete it.
  2. DRY: one set of assertions for both modes. if env.is_mock() should be rare. The same assertion must pass whether the backend is mock or live. Mock is a CI convenience (API key substitute), not a separate test suite.
  3. Mock does not count toward coverage. The 90% target counts features exercised through real user journeys. A mock-only test that never runs live is not coverage — it's a unit test with extra steps. Every test MUST pass in live mode to earn coverage credit.
  4. Agent trigger verification. In live mode, verify the LLM selected the correct tools — not just that scode can execute them. expect("read_file") in the PTY output proves the model's tool-call decision, not just the execution path.
  5. Disk verification is the strongest assertion. After a write/edit, check the actual file on disk. This works identically in both modes and catches real bugs that output assertions miss.
  6. Live tests run serially. TestEnv holds a LIVE_SERIAL mutex in live mode — concurrent scode processes hitting the same API key cause rate-limit failures. Mock tests stay parallel.
When to use mock vs live vs no-API

Not every PTY test needs an API call. Categorize by what the test exercises:

CategoryBackendExample
No-API — pure CLI behavior (slash commands, subcommands, tab completion, system-prompt output)Mock is fine (provides auth config so the REPL starts, but never calls the API). No SCODE_TEST_BACKEND=live needed.memory_tab_completion, memory_entries_injected_into_system_prompt
LLM-dependent — model must choose tools, generate output, make decisionsLive required (SCODE_TEST_BACKEND=live). Mock mode: skip with message. These are the tests that earn coverage credit.memory_write_read_forget_workflow, multi_tool_roundtrip

Rule of thumb: if the test never waits for a model response (no expect after a user message that requires LLM tool use), it's no-API. Use mock. Don't force live mode for tests that don't touch the model — it wastes API tokens and adds serial-lock contention.

Test depth convention (in the Notes column of coverage tables):

  • +error — error path also covered (permission denied, file not found, etc.).
  • No tag — happy path only. When adding tests, check if an error-path test exists; if not, add one.

Test layer policy: PTY is the bar, unit tests are forbidden

Most engineering orgs reach for a three-layer pyramid by default (unit / integration / e2e). We deliberately do not. The product owner's empirical observation, drawn from a year of agent-driven development across multiple projects: unit tests have not caught a single real bug in that window. The bugs that shipped were caught either by the e2e harness or by humans using the product. Unit tests in this codebase have so far functioned as overhead — pure re-statements of the implementation, dragged alongside every refactor, paying for themselves in neither caught bugs nor refactor confidence.

So the policy for sudocode is:

LayerStatusWhen to write
PTY (pty-expect) Required CI gate; the only layer that counts toward the 90% e2e coverage metric Every user-facing feature
Integration (mock harness) Optional — debug accelerator, not a gate When a PTY failure's root cause sits behind a wide blast radius (LLM transport, runtime state machine, ACP wire) and shrinking the bisection distance is worth the file
Unit Forbidden by rule Don't write them. PRs that add unit tests will be asked to remove them. There is no "exceptional case" carve-out — the rule is the rule, intentionally bright-line so it doesn't erode case by case.
Why we trust this — fast triage replaces fine-grained tests

The defense most often given for unit tests is failure attribution: "PTY tests just say the REPL is wrong somewhere; unit tests point at a line." That defense collapses if PTY failures are themselves fast to triage. Our substitute is trace quality:

  • Structured logs across runtime / API / tool dispatch so a PTY failure carries enough breadcrumbs to walk back to the offending layer in minutes, not unit-test-precision seconds.
  • Mock harness available on demand for the wide-blast-radius cases (it stays opt-in, not a CI gate, but it lives in the repo).
  • The agent (sudocode itself) doing the debugging — the assumed developer workflow on this project is agent-assisted, and an agent reading a structured trace converges as fast on root cause as a human reading a failing unit test name.
What we give up — explicit, not hidden
  • cargo test --workspace wall-clock no longer benefits from a unit layer. The runtime is dominated by PTY tests. We accept this; sub-second inner-loop feedback is not the workflow we are optimising for.
  • Rare error branches (panic recovery, edge-case config parse errors) get exercised only when a PTY scenario reaches them. Branches the PTY scenarios never touch will silently rot. We accept this; if those branches never trip in real use either, the rot is harmless, and if they do trip we will see it as a user-visible failure and write a PTY scenario.
  • Internal refactors lose fine-grained behaviour verification. We accept this; PTY tests describe the contract the user actually cares about, and refactors that don't move the user-visible contract are exactly the ones unit tests would have re-stated rather than verified.

This is a deliberate, owner-blessed deviation from the standard testing pyramid, and the rule is bright-line on purpose. "We'll allow it just this once because the input space is combinatorial" is exactly how the unit layer reappears, accumulates, and starts costing refactor velocity again. So: no unit tests, no exceptions. If a future maintainer wants to reopen this, the burden is a concrete bug that a unit test would have caught and a PTY scenario would not — and a separate ROADMAP edit to change the rule before the test lands, not the test landing first.

Mirrored from the original sudowork-side coverage tracker (sudowork/docs/scode-e2e-coverage.md) and status reset for the new sudocode-side harness. The previous sudowork-YAML coverage drove features through the sudowork UI (mouse / Electron / ACP); the new infra exercises them inside the sudocode crate itself, so coverage starts fresh and is tracked here as the single source.

Status values (use data-status attribute for programmatic access):

  • Covered — PTY test exists on main that passes in live mode (SCODE_TEST_BACKEND=live). Mock-only tests do not earn Covered status.
  • Gap — needs a test.
  • N/A — out of scope for scode itself.

Additional data-* attributes on Status cells:

  • data-live-verified="yes" — test has been run and passed in live mode against a real API. AI agents: when marking a feature Covered, also set this attribute after verifying live mode passes.
  • data-live-verified="no-api" — test does not call an API (e.g. system-prompt subcommand), so live/mock distinction is N/A.
  • Absent — test exists but live mode not yet verified.

Hacker priority (the new column):

  • must-have — the hacker workflow blocks without it: invocation, streaming feedback, mid-flight cancel, bash/edit/grep, /commit, /pr, /resume, --output-format json.
  • nice — L2-deferred surface (LSP, sub-agents, cron, MCP plugins): real features, not on the critical path for e2e ≥ 90%.
  • N/A — sudowork-UI / credential-injection concerns; never enters scode's coverage denominator.

Today's stack (sudowork client + sudocode + this preflight chain) eventually surfaces a "your image is too large, please compress" message to the user when payloads exceed transport / preflight caps (sudowork transport cap 25 MB, sudocode preflight cap 5 MB, see commits 0d39128 / f44db9b4). Asking the human to manually compress / split images is the wrong contract — any modern AI client (CC, Cursor, ChatGPT, Gemini) handles this transparently. The proposed fix has two layers, both server-side:
1. Auto-downsample at the input boundary (independent of vision-capability of the chat model): always run image preflight on receive, accept anything that the image crate can decode, downsample to a known-good size for the active model's vision quota (or to 1 MB if the active model has no vision quota). User never sees a size error from this layer.
2. Auto-route to VLM when the active chat model doesn't support vision (today this returns "当前模型不支持图片分析"). Send the image to a separate VLM call (callChatCompletionsWithImage already exists for the /image analyze slash command — promote that path to a transparent fallback used whenever a chat-attach image hits a non-vision chat model), get a text description, splice it back into the user turn as "[Image #N: <description>]". User never sees a model-support error from this layer.
Net: the user-facing "size error" + "model-doesn't-support" messages should both disappear. The PR-F sudowork-client size cap (IMAGE_ATTACHMENT_TRANSPORT_CAP_BYTES = 25 MB + the soft "临时上限" stopgap message in src/common/image-attachment-guard.ts) is the explicit placeholder until this lands.

Permission system — nexus rebac observation

The current permission system is two files, two structs:

  • PermissionPolicy (runtime/src/permissions.rs) — rule evaluation: allow/deny/ask rules, mode comparison, prompter interface.
  • PermissionEnforcer (runtime/src/permission_enforcer.rs) — tool-level gate: wraps policy, adds workspace boundary + bash command classification.

This is ABAC-style (path-prefix matching: "this path is allowed for this tool"). It's correct and zero-overhead for single-user CLI. Nexus's rebac service is relationship-based ("agent A has write permission to resource R via role"). The two solve different problems:

  • Current ABAC — sufficient for single-agent CLI. String prefix match at policy construction time, not per-call.
  • Nexus rebac — needed when multi-agent federation lands (agent A can read agent B's memory, but not agent C's). Not worth migrating before that.

Decision: keep current system until multi-agent use cases require relationship-based access control. At that point, evaluate whether nexus rebac can serve as the backing store for PermissionPolicy rules, or whether the two layers remain separate (rebac for cross-agent, ABAC for intra-agent tool gating).

Slash command typeahead — replace rustyline Tab completion

CC shows a dropdown suggestion list the moment the user types /, without requiring Tab. scode currently uses rustyline's Completer trait — functional (Tab shows options, partial prefix completes), but requires an explicit Tab press.

Plan: replace the current rustyline Tab completion with an inline typeahead that renders suggestions below the prompt as the user types. Delete the existing SlashCommandHelper / Completer implementation in input.rs — the new behavior fully supersedes it.

CC reference: useTypeahead hook (src/hooks/useTypeahead.tsx, 1384 lines). Heavy in code because it handles all input modes (slash commands, @ file index, shell history, Slack channels, path completion, session resume). The slash-command portion is ~50 lines of string prefix matching — runtime overhead is negligible. The rest of the hook's scope is out of scode's current surface.

PTY test impact: memory_tab_completion sends /mem + \t to trigger rustyline completion. When the typeahead replaces rustyline, this test must change: send /mem → expect suggestions rendered inline → select → verify execution. The Tab send becomes irrelevant.

Priority: polish. Current Tab behavior is adequate. Track for later.

No alternate-screen / full-screen TUI mode — by rule

Earlier drafts of this plan included a "Phase 6 — full-screen TUI" sub-goal: a --tui flag wired to ratatui behind a full-tui cargo feature, with split-pane layout, scrollable conversation view, mouse support. This is forbidden by rule. sudocode does not adopt an alternate-screen mode, does not depend on ratatui, and will not ship split-pane or in-CLI multi-agent dashboards.

The reasoning lines up with sudocode's sole-audience positioning (hacker — see What sudo code is) and the copilot-worker topology sudocode plugs into (see Collaboration as a unit below):

  • Hackers already have the TUI stack. VS Code shows diffs and syntax. tmux owns status bars and pane splits. lazygit / ranger own focused TUI nav. sudocode fighting tmux for split-pane real estate is exactly what claude-code does to push non-coders into coding — and exactly what pisses hackers off.
  • Alternate-screen kills scrollback. Hackers run scode in a VS Code terminal, in a tmux pane, in an SSH session. Quitting an alternate-screen app wipes the buffer; the wheel / Ctrl+B [ / Cmd+K muscle memory dies. This is one of the concrete things claude-code's pager / fullscreen prompts do that hackers route around.
  • Alternate-screen breaks copilot-worker attach. In the copilot-worker topology, an operator (or hydra-style UI) attaches to a worker's terminal to inspect live state. A worker that ran in alternate-screen would hijack the operator's terminal on attach and erase its scrollback on detach. Inline scrollback IS the affordance that makes attach/detach safe.
  • Pipe-friendliness is load-bearing. scode "plan X" | tee /proc/p_worker/chat-with-me only works when scode writes to a regular tty / pipe. Alternate-screen makes scode a black-box program, breaking unix composition. The same primitive that lets a hacker pipe scode output around is what lets sudocode participate as a unit in nexus-vfs chat-with-me streams.

What is in scope, and already shipped in PR #180 (2026-06-05): inline ANSI polish — colored diffs, syntax-highlighted tool results, post-turn tool timeline, box-drawn permission prompts, per-turn status line, $PAGER shell-out, fuzzy session picker, tab completion. These are inline rendering improvements; they do not enter alternate-screen, do not break scrollback, do not require new dependencies, and are friendly to attach/detach workflows. Future inline polish welcome under the same constraints.

Reopening this rule requires the same shape as the no-unit-tests rule (Goal 1, "Test layer policy"): a concrete collaboration scenario where inline rendering demonstrably fails to serve a hacker workflow, and a separate ROADMAP edit to change the rule before any ratatui / alternate-screen code lands. "It would be nicer" doesn't qualify.

Inline rendering design principles

  1. Inline REPL only. No alternate-screen mode — see the bright-line rule above. Scrollback in the host terminal (VS Code, tmux, SSH) stays load-bearing.
  2. Every formatting function takes &mut impl Write so it stays testable without a terminal, and so the same code path serves --output-format json, ACP, and pipe consumers.
  3. Rendering works incrementally; the response stream renders as it arrives.
  4. Terminal control goes through crossterm uniformly; raw ANSI escape codes stay out of the codepath.
  5. No ratatui dependency. Even behind a feature flag — its presence in Cargo.lock is itself the slippery slope toward the alternate-screen path the rule above forbids.

Candidate · Plan mode as active compaction

Plan mode is active, prospective context reset — the user has decided to bound the next thought. They are semantically distinct primitives and we should treat them that way.

Sketch (needs CCB validation pass + design write-up before priority):

  • One session = one plan. Fixed file path per session (e.g. <session-id>.plan.md on the session-scoped filesystem). No slug, no per-plan identity confusion — CC's slug-based plan files cause the LLM to ask "which plan?" mid-conversation, which the bounded one-per-session model eliminates.
  • Entering plan mode triggers active compaction. Agent reads the current plan, the runtime drops history below the plan + system prompt, plan becomes the focused context. The plan IS the summary, so no summarization loss the way /compact introduces.
  • Inside plan mode, the agent edits the plan incrementally. Delete items marked 100% done. Append new items as they're decided. Revise pending items (don't delete future items — can re-word, can re-scope, can re-order). The agent does this work; the user supervises.
  • Multiple plans = multiple sessions. No "switch plan within one session" UI — that's how CC's plan-file confusion happens. Session-bounded plans stay unambiguous.

This is a candidate, not active work. CCB validation against CC's plan mode comes first (per the parity standing rule), then a design write-up, then a priority decision. Logged here so the insight isn't lost to discussion-only memory.

Deferred · ! bash mode

An earlier active design write-up for a CC-style !cmd prefix that bypasses the LLM round-trip and dispatches the rest of the line to a shell (e.g. !ls, !git status, !cd path) lived here in full detail (~250 lines covering Bash provider abstraction, pipe rearrangement under !, permission validator behavior, ConPTY handling, CCB validation against @91cffe16). Removed and deferred indefinitely on 2026-06-14.

Why deferred. Empirically, !cmd shortcuts don't pay productivity for the audience this ROADMAP serves (see README § Who this is for). Heavy users bundle: a single long prompt to the agent covering multiple needs, with the agent doing the shell calls inside its tool loop. Inserting an !ls shortcut as a one-off interaction breaks that rhythm — the user pays the cost of leaving the agent thread to run one command, then returns to the prompt. The savings are negative.

The full design lives in git history (preserved by the no-squash merge policy). Reopen if a real heavy user asks — same friction gate as publish_to_shareone above.

Goal 4 · Be a good unit in the copilot-worker topology

The positioning — sudocode as one agent unit among peers on the nexus VFS chat-with-me plane, interchangeable with Claude / Codex / humans at the mailbox primitive — and the three diagrams (topology · one-binary-three-modes · hydra evolution) live in README.md § Position in the larger picture. That's where a hacker should first encounter the framing, alongside the rest of the philosophy.

This goal section holds the engineering plan: what sudocode commits to implementing against the nexus protocol, what is deliberately not in scope, and the work sequencing. The protocol SSOT itself lives upstream in nexus integration architecture (public, Apache-2.0, joint-owned with the nexus team; SSOT at nexus/docs/architecture/nexus-integration-architecture.html). sudocode does not vendor it.

What sudocode commits to — implementation contract

  1. Spawn surface. A sudocode session may be started by an external orchestrator through ManagedAgentService.start_session_v1({agent_id, repos, model, owner_id, zone_id}). The returned session_id IS the AgentRegistry pid; sudocode does not invent its own. The same binary stays runnable as a standalone CLI without nexus — the FsBackend trait abstracts this.
  2. FsBackend trait. All filesystem access in the sudocode runtime routes through one FsBackend trait with three impls: KernelFsBackend<Kernel> (in-process kernel syscalls when running inside sudocode-host), StdFsBackend (host std::fs for the standalone CLI), NexusVfsFsBackend (gRPC to a remote kernel from an edge / dev CLI). No code path bypasses the trait — not session jsonl, not config, not the workspace-bounded file_ops helpers.
  3. Mailbox read / write. A sudocode session watches its own /proc/{self_pid}/chat-with-me via sys_watch for incoming prompts; it writes responses to the conversation peer's chat-with-me using sys_write. The on-disk envelope's from field is stamped by the kernel (MailboxStampingHook); sudocode does not forge it.
  4. Peer addressing inside a workspace. An agent staged at /proc/{pid}/workspace/... writes chat-with-me relative to its cwd — that path is a DT_LINK back to the workspace owner's mailbox. sudocode honours the convention; it does not bypass the link to find the owner pid out-of-band.
  5. Workspace boundary. A write into another agent's workspace/ returns EPERM from WorkspaceBoundaryHook with a structured payload telling the LLM where to write instead (the chat-with-me link). sudocode surfaces this EPERM verbatim back to the LLM rather than catching and retrying — the path layout is the SSOT for permissions and the error message is the teaching channel.
  6. Session state placement. Inside sudocode-host, session jsonl transcripts live at /agents/{name}/sessions/<workspace_hash>/<session-id>.jsonl on the kernel VFS, not on the host filesystem. Booting a worker reads its profile + the requested --session-id transcript from that VFS path; the same path supports resume.
  7. Lifecycle. Out-of-band termination (SIGTERM / SIGKILL / orphan reap from AgentRegistry) must drop the session cleanly — flush in-flight chat-with-me writes, close streams, release locks. No long-running in-memory state that doesn't survive a host restart; the --session-id transcript is the SSOT.

What is NOT sudocode's job here

  • Orchestration. Deciding which workers exist, when to spawn, when to cancel — that is the orchestrator (sudowork / hydra / a script). sudocode receives spawn calls; it does not invent them.
  • Multi-agent UI. Dashboards, worker-vital sidebars, attach buttons — sudowork (Feishu / web) and hydra (VS Code sidebar) own that surface. sudocode stays the unit. This is the deepest reason the alternate-screen TUI rule (Goal 3) is bright-line: a dashboard inside the unit is a category error.
  • Cross-instance transport. Raft replication of DT_STREAM, federation Raft cluster topology, Matrix C-S adapter for stock chat clients — all kernel / nexus-services concerns. sudocode reads and writes through FsBackend and doesn't care whether the stream is same-host or replicated.
  • Audit. AuditHook is a kernel POST-write hook fired by every sys_write; sudocode needs to do nothing to participate.

Sequencing

The work falls into a few sequenced chunks; each ships its own PR. None of this depends on Goal 3 inline polish; the two streams proceed in parallel.

  1. FsBackend trait + StdFsBackend. Audit every std::fs call site in the sudocode runtime, route through the trait. No behavioural change to standalone CLI; this is a refactor that opens the door for the other two backends. DONE — PR #120.
  2. NexusVfsFsBackend. gRPC client against the nexus NexusVFSService.Call surface. First end-to-end validation: a standalone scode pointing at a remote nexus serves its session jsonl from the kernel. PARTIAL — code exists, lacks e2e validation.
  3. KernelFsBackend + sudocode-host binary. The binary edge that links nexus kernel + services in-process and monomorphises SpawnTask<Kernel>. This is where the dependency edge sudocode → nexus is realised. PARTIAL — KernelFsBackend implemented, sudocode-host binary not yet created.
  4. Mailbox read/write loop. Inside the managed runtime, replace stdin/stdout prompts with sys_watch on /proc/{self_pid}/chat-with-me and sys_write on the peer's mailbox. The runtime surface to the LLM stays the same. DONE — PR #119, sys_watch condvar.
  5. WorkspaceBoundaryHook EPERM round-trip. Surface the kernel's structured EPERM verbatim into the LLM conversation; confirm by PTY scenario that the LLM learns the chat-with-me redirect without system-prompt edits. NOT STARTED.

Feature-parity standing rule: sudocode ↔ sudowork dedup

When adding a feature to sudocode, decide whether to remove it from sudowork. If both products expose the same capability (file operations, git workflow, session management, tool execution), UX consistency becomes impossible to maintain across two codebases in two languages. The standing rule:
  1. Before shipping a feature in sudocode, check whether sudowork already has it (renderer-side or worker-side).
  2. If yes, decide now — not later — which product owns it going forward. "Both" is the wrong answer unless they serve genuinely different user journeys (e.g. sudowork's GUI file tree vs sudocode's CLI read_file).
  3. If sudocode becomes the owner, open a sudowork PR to deprecate or remove the duplicate. Ship both PRs in the same cycle.
  4. If sudowork keeps it, don't add it to sudocode — route through sudowork's existing surface instead.
The goal is one SSOT per feature, not two implementations that drift.
