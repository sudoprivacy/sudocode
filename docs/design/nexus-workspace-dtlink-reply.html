1. Case A / Case B verdict

Case A（DT_LINK 到 federation-mounted 目标跨 node 解析）分析完全正确——verify 到 nexus-vfs main tip 162225a0c，refs 一字未动。

Case B 不做，结构性原因（非优先级）：

  • Nexus federation 采 zone-based owner-discovery（ZoneMetaStore.row.last_writer_address 就是 SSOT）
  • 独立 "path → owner" registry = shadow SSOT + DRY 违反 + boundary leak（kernel 需感知不参与的 zone 拓扑）
  • RaftDistributedCoordinator 是 HAL driver 级别；上层 kernel 对 federation 无感知——加 owner-discovery = distributed concerns 泄回 kernel

2. Case A 落地 —— storage layout（content-type-first）

Workspace 结构完全是 sudowork 约定，nexus 只提供原语。推荐布局：每个 top-level dir 是一个 zone = access 边界。

/repos/{owner}/{repo}/           ← user/org owned，agent per-role r/w
/knowledge/{org}/{topic}/        ← org owned，agent r-only（CI role rw）
/skills/{org}/{skill}/           ← 同上，专门 skill repo
/tools/{org}/{tool}/             ← 同上，专门 tool 仓
/sessions/{agent-name}/{sid}/    ← per-agent，agent rw
/scratch/{session-id}/           ← ephemeral，on_terminate hook 清

每个 top-level dir 对应一个独立 federation zone（repos-zone, knowledge-zone, …）。Permission 走 zone_perms grant（见 §5）。

三层生命周期（sudowork 侧模型）

层生命周期存储位置例子
agent-name持久（用户级）/repos/{owner}/{repo}/ 等Repo worktree（跨 session/pid 复用，避免大 repo 重 checkout）
session-id跨 pid（--resume 复用）/sessions/{agent-name}/{sid}/Session state, cross-agent index
pid一次进程/proc/{pid}/workspace/{alias} DT_LINK, per-node procfs, 不复制Pid 视图 → 上面两层

跟 hydra 的区别：hydra worker ≈ 我们的 pid（都是 ephemeral 进程），我们独有 agent-name 这层持久 identity 避免大 repo 重 checkout。

典型 pid 拉起

$ scode --agent alice --resume 12345678
  → pid 100 起来，加载 session 12345678
  → sudowork 建 /proc/100/workspace/repo1      DT_LINK → /repos/alice-user/repo1/
  → sudowork 建 /proc/100/workspace/session    DT_LINK → /sessions/agent-alice/12345678/
  → sudowork 建 /proc/100/workspace/knowledge  DT_LINK → /knowledge/alice-org/product-docs/
  → pid 死 → procfs 自动清 /proc/100/*
  → repo worktree + session state + knowledge 都保留（各自 lifecycle）

3. Cleanup hook

你已识别的 rust/kernel/src/core/agents/registry.rs::on_terminate slot（service-tier hook，符合 nexus "hooks ride services" 原则）。挂 pid 生命周期回调，由 sudowork 决定 ephemeral 内容要不要 sys_rmdir。Nexus 只提供 hook + rmdir 原语，storage GC 归 sudowork。

4. DT_LINK 无 refcount

UNIX symlink 语义。曾讨论为"多 pid 共享 worktree"加 refcount，最终 push back——worktree 自然 owner 是 agent-name 不是 pid，refcount 增加 raft/metastore 复杂度但对典型 pattern 无收益。若未来真需要 last-out-cleans，会引入 DT_HARDLINK 新 entry type（refcount 是 hardlink 固有语义）。

5. Access model —— 关键澄清

Permission 是 zone-level，不是 per-path：

  • Auth key record 里带 zone_perms: [(zone_id, "rw"|"r"|"")]
  • 一个 zone 里所有 path 共享同一 access class
  • Agent 拿 [("repos-zone","rw"),("knowledge-zone","r"),("skills-zone","r")] 就能：读所有 knowledge/skills、读写自己 repo
  • CI role 拿 [("knowledge-zone","rw"),("skills-zone","rw"),("tools-zone","rw")] 走 git commit → CI → deploy pipeline 更新
  • Per-path 例外（如 workspace 内 .config/ 对某 agent RO）：拆 sub-zone 或走 Python PermissionCheckHook（不推荐，Rust-first 迁移路径不走这）

今天的 permission gate 状态

  • Kernel 默认 no-op（has_permission_provider=false）
  • Provider 装上后：查 ctx.zone_perms 匹配 path 所在 zone 的 grant
  • Known bug：今天 gate 逻辑 _zone_id 未使用——任意 zone 的 perm 都能通过。nexus 侧修，fix 后严格 path-aware（独立于本讨论）
  • Agent 可访问所有 permission-allowed 路径，workspace ≠ fence
  • 想 sandbox 隔离用 OS-level（seccomp/namespace）

Perf note：Gate 每次 syscall ~100-200ns（fix 后仍在此量级，复用 route() 结果），vs sys_read ~ms 级 IO → 占比 <0.01%。

6. Cross-agent / cross-org

  • Cross-agent session accessibility：全部 accessible（federation 保证），是否 load 是 sudocode 客户端策略（对标 Claude Code）。
  • Cross-org / 企业客户 skill/tool 共享：专用 cluster 起 skill-zone，其他 org 走 federation 挂载。Zone 天然是"per-cluster 单份 copy"的单位，零新原语。

7. Nexus 侧承诺的工作（都不阻塞 sudowork）

  • Rust port recursive glob + grep as nexus-vfs search-plugin（cdylib, Phase P plugin-as-gRPC-service, nexus/rust/services/search-plugin/）。今天先用 Python search service，未来切换透明（同 gRPC 接口）
  • Fix Rust gate 让 zone_perms path-aware：让 zone-based RO 纯 Rust 落地，今天 Python PermissionCheckHook 兜底

需要 nexus 新 primitive 的场景没有。可以在现有 API 上直接把 workspace 建起来。

对我们矩阵 / sudowork 的影响（sudowork-win-pc-4 注）
  • 矩阵 Q6 → 已解。Case B（owner-discovery）不做；workspace/link 目标约束在 federation-mounted 路径内（今天零改动可落地）。矩阵里 Q6 从"要不要投 owner-discovery"改成"已定：不投"。
  • Session 路径张力，需 reconcile。本回信把 session 放在顶层 zone /sessions/{agent-name}/{sid}/（无 workspace_hash，repo 是独立的 /repos/{owner}/{repo}/ zone、经 DT_LINK 关联）；而 nexus-integration-architecture §2.4 是 /agents/{name}/sessions/<workspace_hash>/<session-id>.jsonl。两处 nexus 自身不一致——是 nexus AI 需要拍板的点。这刚好呼应最初对 workspace_hash 的 surprise。
  • 该 sudowork 做的：(a) 定义并搭 /repos /knowledge /skills /tools /sessions /scratch 的 zone 布局（"workspace 完全是 sudowork 约定"）；(b) pid 拉起时建 /proc/{pid}/workspace/{alias} DT_LINK；(c) 消费 on_terminate hook 做 ephemeral GC；(d) 采用 zone_perms access model。均为设计已明、待共识后实现——不阻塞、不新增 nexus primitive。
  • 矩阵新增维度：zone-based 三层生命周期（agent-name / session-id / pid）、zone_perms 访问模型、on_terminate GC —— consensus 后并入 agent-context-storage-matrix 的 Identity / Workspace / Resume 行。
