Agent Context / Image / Profile — Unified Storage Matrix

A 1:1 mapping of Claude Code's on-disk context model onto the Nexus /agents/{name}/+/proc/{pid}/ design, tracked through to sudocode & sudowork. Draft for consensus — not yet merged into the doc.

The one principle. Nexus splits an agent the way an OS splits an executable on disk from a running process: image / static / class layer = /agents/{name}/ (config · prompts · skills · memory · sessions — persistent, Metastore, the addressable identity; one name → many pids) vs runtime / dynamic / object layer = /proc/{pid}/ (status · agent-link · chat-with-me · workspace/ — ephemeral, stamped at start_session, reaped on exit; pid = AgentRegistry SSOT = the session id).

Claude Code is a flat special case of this: its ~/.claude/projects/<sanitized-cwd>/ collapses image+runtime into one cwd-keyed folder — one implicit agent per project. Everything CC persists therefore has a home in our two-layer model. This table is that map, and the SSOT for the dup that sudowork & sudocode carry today and must merge long-term.

Open questions (for consensus)

Q1 Adopt agent-name as primary key (Nexus superset) as canonical — a real shift from CC/scode's cwd-primary? (§ Identity)
Q2 Memory tiers: agent-memory (/agents/{name}/memory/) + project-memory. Given workspace can DT_LINK multiple repos, where does project-memory live — at the workspace's primary mount root (≈ launch path)? Which of the 6 missing memory rows do we build, and in what order?
Q3 Cross-agent session search via DT_LINK /agents/{name}/sessions/ → shared /agents/sessions/? (depends on Q6)
Q4 Oversized tool-output offload (CC's tool-results/ + content-replacement) into scode — adopt? Path under /agents/{name}/sessions/<hash>/tool-results/?
Q5 Subagent: full-transcript fork parity (vs cache-prefix)? Move subagents to nexus-managed pids (sudocode → MAS::start_session)? Storage → agent-<id>.jsonl+.meta.json parity?
Q6 Workspace & DT_LINK — confirm default-mount of the launch path (≈ project root) as workspace[0] + multi-repo via workspace/{alias}→DT_LINK→mount_path. Cross-node resolved: a link to a federation-mounted target already resolves cross-node today, zero change (the follow is a syscall re-entry that inherits DT_FILE's try_remote_fetch/last_writer_address; DT_STREAM/PIPE similarly "just work" as replicate-everywhere). The only new work is a link to a target on a node with no local mount/zone → needs genuinely-new owner-discovery ("who owns path X"), which no primitive provides (read-miss path contractually refuses peer fan-out). Decision: constrain workspace/link targets to federation-mounted paths (ship now, zero change), or invest in owner-discovery for arbitrary remote targets (separate design)?
Q7 Persistent-assistant pointer: adopt CC's mtime-heartbeat + cross-worktree-freshest for the standalone/offline scode case (AgentRegistry is SSOT when nexus present)?
Q8 Doc drift: nexus-integration §3.3 says the mailbox stamp hook is "owned by ManagedAgentService" — code moved it to the kernel-tier a2a substrate (armed at boot, for all writers incl. matrix humans). Update §3.3.
Q9 Migration & dedup sequencing — priority/order of: (a) wire scode's dormant nexus KernelFsBackend seam (flat .scode/sessions/ → /agents/{name}/ layered); (b) collapse sudowork messages → a view over the engine JSONL keyed by acpSessionId (keep only UI-unique fields); (c) build the missing memory features. Which first?
